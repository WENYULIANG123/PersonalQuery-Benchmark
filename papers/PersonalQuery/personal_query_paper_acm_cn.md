# PersonalQuery：基于语言风格对齐的接地气个性化搜索查询生成

## 摘要

个性化搜索系统传统上依赖用户画像或协同过滤，往往无法捕捉细粒度用户偏好和沟通模式。我们提出PersonalQuery，一个多阶段流水线，通过从历史评论中提取细粒度用户偏好并将查询语言与个人写作模式对齐，生成“接地气”的个性化搜索查询。我们的方法集成了基于大语言模型（LLM）的偏好提取、16维语言风格分析、迭代式特征感知优化，以及基于CNN的拼写难度预测以实现靶向噪声注入。在Amazon产品评论上的实验评估表明，与基线方法相比，PersonalQuery在语义相关性和风格真实性方面均取得显著改进。人类评估证实LLM与人类判断之间存在强相关性（Spearman ρ = 0.82）。我们的关键方法论贡献是提出基于双重差分（DiD）评分的偏差感知评估框架，揭示了传统指标对高用户-大众相似度用户的个性化收益低估了高达48%。

**CCS概念**：• 信息系统 → 个性化；查询表示；• 计算方法 → 自然语言生成

**关键词**：个性化搜索、查询生成、用户建模、语言风格迁移、大语言模型

## 1. 引言

### 1.1 动机

电商搜索系统充当用户与庞大产品目录之间的主要接口。传统搜索界面对所有用户一视同仁，无论个体偏好、专业水平或沟通模式如何，均返回相同结果。这种“一刀切”的方法无法捕捉用户在表达需求和评估产品方面的丰富异质性。

专业手工艺人搜索"3mm德式玻璃闪粉，80目纹理"与初学者询问"闪亮的工艺闪粉"的沟通方式截然不同。用户的搜索查询不仅反映其信息需求，还反映其沟通风格、技术娴熟程度和个人偏好。

### 1.2 研究挑战

我们将**接地气个性化**定义为生成满足以下条件的搜索查询：

1. **语义忠实**：行为数据提取的用户偏好
2. **风格真实**：反映个人写作模式
3. **行为现实**：包含拼写模式等自然变体

实现接地气个性化需解决以下挑战：
- **细粒度偏好提取**：超越粗粒度类别，达到属性-情感对级别
- **风格建模**：捕捉句法复杂性等超越词汇的语言模式
- **可控生成**：平衡语义准确性与风格对齐
- **现实变体**：复现用户特有的拼写倾向等特质

### 1.3 贡献

我们提出PersonalQuery，一个通过以下方式解决这些挑战的综合流水线：

1. **多阶段处理流水线**：将原始用户评论转化为具有可控风格属性的个性化搜索查询（第3节）

2. **接地气画像生成**：从行为信号综合技能水平、使用场景、情感配置和价值维度（第3.2节）

3. **迭代式风格优化**：通过特征感知提示逐步将生成的查询与目标风格画像对齐（第3.4节）

4. **基于CNN的拼写难度模型**：实现保持语义保真度的靶向噪声注入（第3.5节）

5. **偏差感知评估框架**：使用双重差分（DiD）评分隔离真正的个性化价值，修正基线相似度效应（第4.2节）

## 2. 相关工作

### 2.1 个性化搜索与推荐

个性化搜索系统已从简单用户画像发展到复杂神经方法。早期工作集中在基于用户历史的点击率预测和结果重排序[1]。近期方法采用深度学习建模用户偏好，但通常在结果层面而非查询生成层面操作[7]。我们的工作区别于现有方法，生成匹配用户特定语言风格同时保持语义意图的查询。

### 2.2 用户偏好提取

从文本中提取细粒度偏好一直通过基于方面的情感分析和观点挖掘进行研究。传统方法依赖手工特征和词典，现代方法利用预训练语言模型[4]。我们的方法通过精心设计的提示工程从LLM提取结构化属性-情感对扩展了这项工作。

### 2.3 语言风格分析与迁移

风格迁移已应用于正式程度调整、情感修改和作者模仿等多个领域[5]。基于特征的方法通常依赖手工特征，而神经方法隐式学习风格表示。我们的工作通过使用显式16维特征配置指导迭代式LLM优化来连接这两种方法。我们利用ProfilingUD[3]进行语言无关的语言学分析。

### 2.4 查询生成与扩展

查询生成已用于对话搜索和查询建议。大多数方法关注语义扩展或澄清问题[8]。我们的工作通过生成匹配用户特定语言风格同时保持语义意图的查询来区分于现有方法。

### 3.2 模块1：偏好提取

#### 用户选择标准

我们基于以下标准从Amazon评论数据中选择高质量用户：
- 评论数量在[100, 110]产品范围内
- 产品具有有效元数据（排除"Unknown"类别）
- 每个产品至少5条评论，以确保可靠的偏好推断
- 每个产品至少4条其他用户的评论，用于公开属性提取

#### 基于LLM的偏好提取

对于每个用户-产品对，我们采用GLM-5从评论文本中提取结构化偏好：

```json
{
  "target_user_preferences": {
    "Use Case": [{"entity": "card making", "sentiment": "positive"}],
    "Quality": [{"entity": "adhesive strength", "sentiment": "positive"}],
    "Design": [{"entity": "color vibrancy", "sentiment": "positive"}]
  },
  "public_attributes": [
    {"attribute": "price range", "sentiment": "neutral"}
  ]
}
```

提取保留每个属性的情感极性（积极/消极/中性），实现情感感知的查询生成。

#### 语义去重

为防止属性冗余，我们应用三级去重：
1. **精确匹配**：四元组（asin, category, attribute, sentiment）去重
2. **字符串相似度**：70% Levenshtein相似度阈值
3. **语义聚类**：Sentence-BERT嵌入[9] + 社区检测（阈值0.85）

### 3.3 模块2：画像生成

#### 技能水平估算

我们基于以下标准对用户技能水平进行分类（初学者/中级/高级/专家）：
- 关键词匹配（如"初学者"、"专业"）
- 技术术语检测（如毫米单位、克每平方米纸重）
- 设备/品牌提及（如"Cricut"、"Sizzix"）

#### 使用场景分类

属性使用关键词匹配映射到预定义的使用场景：greeting_cards（问候卡片）、scrapbooking（剪贴簿）、jewelry（首饰）、costume（戏服）、fabric_crafts（布料工艺）、home_decor（家居装饰）、gift（礼物）。

#### 情感配置

我们计算所有偏好的情感分布：
- 积极/消极/中性比例
- 性格特征推断（如"通常满意"、"挑剔且注重细节"）
- 按偏好类别的维度情感分析

#### 画像综合

最终画像提示整合：产品类别、技能水平及指标、主要使用场景及示例、评价风格和性格特征、维度情感洞察、价值优先级。LLM生成80-120字的接地气画像描述。

### 3.4 模块3：风格分析

#### 16维语言特征

我们使用ProfilingUD[3]提取以下特征：

| 类别 | 特征 |
|------|------|
| **长度** | tokens_per_sent, char_per_tok, n_tokens |
| **词汇** | ttr_lemma_chunks_100, lexical_density |
| **词性分布** | NOUN, VERB, ADJ, ADV, PRON, DET, AUX, PART, SCONJ, CCONJ, ADP |

特征归一化到[0, 1]范围用于距离计算。

### 3.5 模块4：查询生成与优化

#### 初始查询生成

对于个性化查询，我们：
1. 选择3个具有情感平衡的属性（最多2个积极/中性 + 1个消极）
2. 生成反映属性情感的第一人称查询
3. 强制25-30字的长度约束

#### 迭代式特征感知优化

优化过程使用多轮生成和靶向指令：

**第0轮**：基础生成，提供风格描述
**第1轮+**：差距驱动优化

对于每轮：
1. 从当前最佳查询提取特征
2. 计算与用户画像的特征差距
3. 为前Top 5差距生成靶向指令
4. 创建带具体调整的优化提示

**候选选择分数**：
$$\text{score} = 0.7 \cdot d_{\text{style}} + 0.3 \cdot d_{\text{semantic}}$$

其中 $d_{\text{style}}$ 是与用户特征向量的余弦距离，$d_{\text{semantic}}$ 是与原始查询的语义距离。

**收敛标准**：
- 同时满足两个阈值：$d_{\text{style}} < 0.3$ 且 $d_{\text{semantic}} < 0.4$
- 无改进：improvement $< 0.02$
- 达到最大轮数（默认：5）

### 3.6 模块5：噪声注入

#### CNN拼写难度模型

我们训练基于CNN的模型来预测单词级拼写难度：

```
输入: [char_indices, handcrafted_features, user_features]
  │
  ├── 字符嵌入 (64维) → Conv1D (128→512) → 自适应最大池化 → 512维
  │
  ├── 手工特征 (50维): 长度、音节、字符n-gram频率、元音比例
  │
  └── 用户特征 (9维): 错误类型频率、历史错误率
       │
       ↓
  拼接 → FC层 (256→64) → Sigmoid → [0,1] 难度分数
```

#### 靶向噪声注入

使用训练好的拼写模型：
1. 对查询中的每个单词评分难度
2. 选择难度最高的Top-K单词
3. 根据用户的错误分布注入错误
4. LLM验证语义保持

用户错误分布（观察到）：
- 同音混淆：35%
- 字母遗漏：25%
- 字母替换：20%
- 字母插入：15%
- 位置调换：5%

## 4. 实验

### 4.1 实验设置

#### 数据集

我们在Amazon艺术、工艺与缝纫类别上评估PersonalQuery：
- **用户**：50个高质量用户，评论了100-110个产品
- **产品**：约5000个带有元数据的独特产品
- **评论**：约150000条评论（目标+其他用户）

#### 实现细节

- **LLM**：GLM-5用于偏好提取、画像生成、查询生成和评估
- **嵌入模型**：sentence-transformers/all-MiniLM-L6-v2
- **语言特征**：ProfilingUD with spaCy后端
- **拼写模型**：PyTorch CNN，50轮训练，Adam优化器（lr=0.001）
- **优化**：最大5轮，每轮3个候选，温度范围[0.4, 0.7]

#### 基线方法

1. **通用查询**：类别+通用属性（无个性化）
2. **仅属性**：个性化属性，无风格对齐
3. **单轮LLM**：无迭代优化的单次生成
4. **基于模板**：带用户属性的填空模板

### 4.2 评估框架

#### 4.2.1 LLM 5规则评估

我们采用基于5个标准的LLM评估（每项1-5分）：
1. **偏好对齐**：查询反映用户偏好？
2. **画像一致性**：查询匹配用户画像？
3. **语义完整性**：涵盖所有关键属性？
4. **自然性**：查询语言自然？
5. **特异性**：查询适当具体？

#### 4.2.2 偏差感知评估：双重差分

评估个性化查询时的一个关键挑战是**基线相似度偏差**：由于目标用户画像（$P_u$）本质上是大众画像（$P_g$）的子集，公开查询（$Q_g$）在评估为$P_u$时天然获得“保底分”，使得难以分离个性化查询的真正增量价值。

我们引入**双重差分（DiD）评分**：

$$\text{Score}_{\text{DiD}} = [S(Q_u, P_u) - S(Q_u, P_g)] - [S(Q_g, P_u) - S(Q_g, P_g)]$$

其中：
- $\Delta_u = S(Q_u, P_u) - S(Q_u, P_g)$：个性化查询的用户偏好程度
- $\Delta_g = S(Q_g, P_u) - S(Q_g, P_g)$：公开查询的基线优势
- $\text{Score}_{\text{DiD}} = \Delta_u - \Delta_g$：隔离的个性化价值

**理论性质**：
1. **零偏差**：当$Q_u = Q_g$时，$\Delta_u = \Delta_g$，故$\text{Score}_{\text{DiD}} = 0$
2. **单调性**：若$Q_u$比$Q_g$捕获更多独特用户维度，则$\text{Score}_{\text{DiD}} > 0$
3. **尺度不变性**：对绝对分数差异稳健，专注相对差异

#### 4.2.3 人类-LLM对齐指标

- **Spearman ρ**：平均分的秩相关
- **Cohen κ**：偏好选择的一致性
- **Recall@Human**：P(LLM选择个性化 | 人类选择个性化)
- **MAE**：LLM和人类分数之间的平均绝对误差

## 5. 讨论

### 5.1 关键发现

1. **接地气画像提高查询质量**：整合技能水平、使用场景和情感配置比仅属性方法产生更真实的查询。

2. **迭代优化是必要的**：单次生成无法捕获细粒度风格特征；多轮优化与特征感知提示是必需的。

3. **16个特征就足够了**：精心策划的风格特征集捕获了必要的语言变化，而没有高维表示的噪声。

4. **靶向噪声注入保持语义**：与随机错误注入不同，难度引导注入在提高真实性的同时保持语义保真度。

5. **LLM评估与人类对齐**：强相关性（ρ=0.82）验证了基于LLM评估的可扩展性。

6. **偏差修正揭示真正的个性化价值**：DiD评分将差异化比从1.8×（原始）增加到2.7×，表明传统评估指标系统性低估个性化收益。

### 5.2 方法论贡献

除查询生成外，我们的工作在方法论上做出两个关键贡献：

**偏差感知评估框架**：我们识别并解决了个性化系统评估中固有的“基线相似度偏差”问题。我们的四种修正方法（DiD、相似度惩罚归一化、向量正交化、对比提示）为不同评估场景提供了互补方法。

**用户-大众相似度分析**：我们的相关性分析揭示了一个重要洞察：偏差修正的价值随用户-大众画像相似度缩放。对于高重叠用户（>0.9），传统评估掩盖了高达48%的个性化价值。

### 5.3 局限性

1. **领域特异性**：评估限制在艺术与工艺；在其他领域的推广需要验证。

2. **用户冷启动**：流水线需要大量评论历史；新用户可能无法受益。

3. **LLM依赖**：质量受底层LLM能力限制。

4. **计算成本**：12阶段流水线结合迭代优化资源密集。

### 5.4 未来方向

1. **跨领域迁移**：调查跨产品类别的画像迁移。

2. **少样本适应**：为历史有限的用户开发轻量级适应。

3. **实时部署**：针对生产延迟约束优化流水线。

4. **多模态个性化**：扩展到语音查询和视觉搜索。

## 6. 结论

我们提出了PersonalQuery，一个用于生成接地气个性化搜索查询的综合流水线。通过提取细粒度偏好、建模16维语言特征以及采用迭代式特征感知优化，PersonalQuery生成的查询既语义忠实又风格真实。实验结果证明相对于基线方法有显著改进，强人类-LLM对齐验证了我们的评估方法论。拼写难度模型使现实变化注入成为可能，同时保持查询语义。我们的偏差感知评估框架使用DiD评分确保此类系统在研究文献中得到适当估值，揭示了传统指标所掩盖的真正个性化价值。

### B.3 DiD评估算法

```python
def compute_did_score(personalized_query, public_query, 
                      user_profile, public_profile):
    """
    计算查询评估的双重差分（DiD）分数。
    
    参数:
        personalized_query: 生成的个性化搜索查询
        public_query: 基线公开查询
        user_profile: 目标用户的画像
        public_profile: 大众群体画像
    
    返回:
        did_score: 量化个性化价值的DiD分数
    """
    # 步骤1：评估所有四个查询-画像组合
    s_uu = llm_evaluate(personalized_query, user_profile)
    s_ug = llm_evaluate(personalized_query, public_profile)
    s_gu = llm_evaluate(public_query, user_profile)
    s_gg = llm_evaluate(public_query, public_profile)
    
    # 步骤2：计算偏好
    delta_u = s_uu - s_ug  # 个性化查询的用户偏好
    delta_g = s_gu - s_gg  # 公开查询的基线优势
    
    # 步骤3：计算DiD分数
    did_score = delta_u - delta_g
    
    return did_score
```
