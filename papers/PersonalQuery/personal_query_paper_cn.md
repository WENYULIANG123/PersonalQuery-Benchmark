# 接地气个性化搜索查询生成：基于语言风格对齐的多阶段流水线

## 摘要

个性化搜索系统传统上依赖简单的用户画像或协同过滤，往往无法捕捉个体用户微妙偏好和沟通模式的细微差别。我们介绍了PersonalQuery，这是一个新颖的12阶段流水线，通过从历史评论中提取细粒度用户偏好并将查询语言与个人写作模式对齐，生成"接地气"的个性化搜索查询。我们的方法集成了基于大语言模型（LLM）的偏好提取、多维度语言风格分析（16个风格特征）、迭代式风格优化（特征感知提示）和卷积神经网络（CNN）拼写难度预测，以实现靶向噪声注入。在Amazon产品评论上的实验评估表明，与基线方法相比，PersonalQuery生成的查询在语义相关性和风格真实性方面都取得了显著改进，人类评估证实了LLM和人类判断之间存在强相关性（Spearman's ρ = 0.82）。

**关键词**：个性化搜索、查询生成、用户建模、语言风格迁移、大语言模型

## 1. 引言

### 1.1 背景与动机

电商搜索系统充当用户与庞大产品目录之间的主要接口。传统的搜索界面一视同仁地对待所有用户，对于相同的查询返回相同的结果，无论个体偏好、专长水平或沟通模式如何。这种"一刀切"的方法无法捕捉用户表达需求和评估产品的丰富异质性。

近年来，个性化搜索的进展主要集中在基于用户历史或协作信号的重新排序上。然而，这些方法忽略了一个关键维度：查询本身。用户在搜索查询中使用的语言不仅反映了他们的信息需求，还反映了他们的沟通风格、技术娴熟程度和个人偏好。一个专业手工艺人搜索"3mm德式玻璃闪粉，80目纹理"与一个初学者询问"闪亮的工艺闪粉"的沟通方式完全不同。

### 1.2 接地气个性化挑战

我们将"接地气个性化"定义为生成满足以下条件的搜索查询：
1. **语义忠实**于从行为数据中提取的用户偏好
2. **风格真实**反映个人写作模式
3. **行为现实**包含自然变化，如拼写模式

实现接地气个性化需要解决几个挑战：
- **细粒度偏好提取**：超越粗粒度类别，到与情感相关的特定属性
- **风格建模**：捕捉词汇之外的语言模式，包括句法复杂性和话语特征
- **可控生成**：平衡语义准确性与风格对齐
- **现实变化**：复制用户特定的怪癖，如拼写倾向

### 1.3 贡献

我们介绍了PersonalQuery，这是一个综合性的流水线，通过以下方式解决这些挑战：

1. **12阶段处理流水线**：将原始用户评论转化为具有可控风格属性的个性化搜索查询

2. **多维度语言特征分析**：使用16个风格特征，包括句法复杂性、词汇密度和词性分布

3. **迭代式风格优化（特征感知提示）**：通过差距分析逐步将生成的查询与目标风格画像对齐

4. **基于CNN的拼写难度模型**：预测单词级易错性，以实现靶向噪声注入

5. **综合评估框架**：结合基于LLM的评估和人类对齐指标

## 2. 相关工作

### 2.1 个性化搜索与推荐

个性化搜索系统已经从简单的用户画像发展到复杂的神经方法。早期工作集中在基于用户历史的点击率预测和结果重新排序。较新的方法采用深度学习来建模用户偏好，但通常在结果级别而非查询生成级别操作。

### 2.2 用户偏好提取

从文本中提取细粒度偏好一直通过基于方面的情感分析和观点挖掘进行研究。传统方法依赖手工特征和词典，而现代方法利用预训练语言模型。我们的方法通过使用精心设计的提示工程从LLM提取结构化属性-情感对，扩展了这项工作。

### 2.3 语言风格迁移与语言分析

语言风格迁移已应用于各种领域，包括正式程度调整、情感修改和作者模仿。基于特征的方法通常依赖手工特征，而神经方法隐式地学习风格表示。我们的工作通过使用显式的16维特征配置来指导迭代式LLM优化，连接了这两种方法。

### 2.4 查询生成与扩展

查询生成已用于对话搜索和查询建议。大多数方法关注语义扩展或澄清问题。我们的工作通过生成匹配用户特定语言风格同时保持语义意图的查询来区分于现有方法。

### 3.2 阶段 0-1：数据准备与偏好提取

#### 3.2.1 用户选择标准

我们基于以下标准从Amazon评论数据中选择高质量用户：
- 评论数量在[100, 110]产品范围内
- 产品具有有效元数据（排除"Unknown"类别）
- 每个产品至少5条评论，以确保可靠的偏好推断
- 每个产品至少4条其他用户的评论，用于公开属性提取

#### 3.2.2 基于LLM的偏好提取

对于每个用户-产品对，我们采用大语言模型（GLM-5）从评论文本中提取结构化偏好。提取提示指导模型识别：

```json
{
  "target_user_preferences": {
    "Use Case": [{"entity": "card making", "sentiment": "positive"}],
    "Quality": [{"entity": "adhesive strength", "sentiment": "positive"}],
    "Design": [{"entity": "color vibrancy", "sentiment": "positive"}]
  },
  "other_users_preferences": {
    "Quality": [{"entity": "durability", "sentiment": "positive"}]
  },
  "public_attributes": [
    {
      "attribute": "price range",
      "sentiment": "neutral"
    }
  ]
}
```

提取保留每个属性的情感极性（积极/消极/中性），实现情感感知的查询生成。

### 3.3 阶段 2-3：偏好分类与数据划分

#### 3.3.1 目标属性 vs 公开属性

我们区分：
- **目标属性**：目标用户独有的偏好
- **公开属性**：其他用户共有的偏好

这种区分使我们能够生成个性化查询（目标聚焦）和基线公开查询（通用偏好）。

#### 3.3.2 语义去重

为了防止属性冗余，我们应用三级去重：
1. **精确匹配**：四元组（asin, category, attribute, sentiment）去重
2. **字符串相似度**：70% Levenshtein相似度阈值
3. **语义聚类**：句子BERT嵌入 + 社区检测（阈值0.85）

### 3.4 阶段 4：接地气画像生成

#### 3.4.1 技能水平估算

我们基于以下标准对用户技能水平进行分类（初学者/中级/高级/专家）：
- 关键词匹配（如"初学者"、"专业"）
- 技术术语检测（如毫米单位、克每平方米纸重）
- 设备/品牌提及（如"Cricut"、"Sizzix"）

#### 3.4.2 使用场景分类

属性使用关键词匹配映射到预定义的使用场景：
- greeting_cards（问候卡片）
- scrapbooking（剪贴簿）
- jewelry（首饰）
- costume（戏服）
- fabric_crafts（布料工艺）
- home_decor（家居装饰）
- gift（礼物）

#### 3.4.3 情感配置

我们计算所有偏好的情感分布：
- 积极/消极/中性比例
- 性格特征推断（如"通常满意"、"挑剔且注重细节"）
- 按偏好类别的维度情感分析

#### 3.4.4 画像综合

最终画像提示整合：
- 产品类别
- 技能水平及指标
- 主要使用场景及示例
- 评价风格和性格特征
- 维度情感洞察
- 价值优先级（维度分数）

LLM生成80-120字的接地气画像描述。

### 3.5 阶段 5-6：风格分析与特征提取

#### 3.5.1 16维语言特征

我们使用ProfilingUD提取以下特征：

| 特征类别 | 特征 |
|---------|------|
| **长度** | tokens_per_sent, char_per_tok, n_tokens |
| **词汇** | ttr_lemma_chunks_100, lexical_density |
| **词性分布** | NOUN, VERB, ADJ, ADV, PRON, DET, AUX, PART, SCONJ, CCONJ, ADP |

特征归一化到[0, 1]范围用于距离计算。

### 3.6 阶段 7：双重查询生成

#### 3.6.1 个性化查询生成

对于个性化查询，我们：
1. 选择3个具有情感平衡的属性（最多2个积极/中性 + 1个消极）
2. 生成反映属性情感的第一人称查询
3. 强制25-30字的长度约束

提示结构：
```
任务：基于用户的属性及其情感生成个性化搜索查询。

产品类别：{category}
用户特定属性及情感：{attrs_str}

要求：
- 反映用户对每个属性的情感
- 长度必须正好是25-30字
- 使用第一人称视角
- 自然且对话式

输出格式：JSON，包含"query"和"word_count"字段
```

#### 3.6.2 公开查询生成

公开查询使用其他用户的3个最频繁属性，遵循相同的格式约束。

### 3.7 阶段 8：迭代式风格优化

#### 3.7.1 特征感知提示

优化过程使用多轮生成和靶向指令：

**第0轮**：基础生成，提供风格描述
**第1轮+**：差距驱动优化

对于每轮：
1. 从当前最佳查询提取特征
2. 计算与用户画像的特征差距
3. 为前Top 5差距生成靶向指令
4. 创建带具体调整的优化提示

#### 3.7.2 候选选择

对于每轮，我们使用组合分数选择最佳候选：

```
score = 0.7 * d_style + 0.3 * d_semantic
```

其中：
- **d_style**：查询与用户特征向量的余弦距离
- **d_semantic**：查询与原始查询的语义距离（1 - cosine_similarity）

#### 3.7.3 收敛标准

优化在以下条件下停止：
1. 同时满足两个阈值：d_style < 0.3 AND d_semantic < 0.4
2. 无改进：improvement < 0.02
3. 达到最大轮数（默认：5）

### 3.8 阶段 9：拼写难度模型

#### 3.8.1 模型架构

我们训练基于CNN的模型来预测单词级拼写难度：

```
输入: [char_indices, handcrafted_features, user_features]
  │
  ├── 字符嵌入 (64维)
  │       │
  │       ▼
  │   Conv1D层 (128→512维)
  │       │
  │       ▼
  │   自适应最大池化 → 512维
  │
  ├── 手工特征 (50维):
  │   - 单词长度、音节数
  │   - 字符n-gram频率
  │   - 元音/辅音比例
  │   - 常见错误模式匹配
  │
  └── 用户特征 (9维):
      - 错误类型频率（同音、遗漏、替换等）
      - 历史错误率
      │
      ▼
  拼接 → FC层 (256→64)
      │
      ▼
  Sigmoid → [0,1]难度分数
```

#### 3.8.2 训练

模型在配对的（正确、错误）标记上训练，难度标签基于：
- 单词频率（越低越难）
- 长度（越长越难）
- 语言陷阱（静音字母、不规则模式）

### 3.9 阶段 10：靶向噪声注入

#### 3.9.1 错误分布建模

对于每个用户，我们计算错误类型频率：
- 同音混淆：35%
- 字母遗漏：25%
- 字母替换：20%
- 字母插入：15%
- 位置调换：5%

#### 3.9.2 难度导向注入

使用训练好的拼写模型：
1. 对查询中的每个单词评分难度
2. 选择难度最高的Top-K单词
3. 根据用户的错误分布注入错误
4. LLM验证语义保持

### 3.10 阶段 11-12：评估

#### 3.10.1 LLM 5规则评估

我们采用基于5个标准的LLM评估：
1. **偏好对齐**：查询反映用户偏好？
2. **画像一致性**：查询匹配用户画像？
3. **语义完整性**：涵盖所有关键属性？
4. **自然性**：查询语言自然？
5. **特异性**：查询适当具体？

每个标准评分1-5，最大总分25。

#### 3.10.2 **基础相似度偏差修正** (新增)

评估个性化查询时的一个关键挑战是**基础相似度偏差**：由于目标用户（$P_u$）本质上是大众的一部分，他们的画像不可避免地与大众画像（$P_g$）存在特征重叠。这导致公开查询（$Q_g$）在评估为$P_u$时天然获得"保底分"，使得难以分离个性化查询的真正增量价值。

我们引入四种互补方法来修正这种偏差：

**方法1：双重差分法**

这是最严谨的统计学修正方法。我们不看绝对分数，而是看交叉对比的相对差异。给定LLM给出的四个基础分数$S(Q, P)$：

```
步骤一：计算查询的"特异性倾向"

个性化查询更偏向用户的程度：Δ_u = S(Q_u, P_u) - S(Q_u, P_g)
公开查询更偏向用户的程度：Δ_g = S(Q_g, P_u) - S(Q_g, P_g) （这个值通常在0左右）

步骤二：计算真实的个性化增益 (DiD Score)

Score_DiD = Δ_u - Δ_g
```

原理解释：这个公式直接抵消了因为$P_u$和$P_g$相似而带来的基础得分。只有当$Q_u$捕捉到$P_u$中异于$P_g$的独特维度时，Score_DiD才会显著大于0。

**方法2：相似度惩罚与归一化**

如果我们先计算出用户画像与大众画像的基础相似度（例如使用Embedding计算余弦相似度Sim(P_u, P_g)），我们可以将其作为一个折扣因子（Discount Factor）引入打分公式中。

修正公式：

```
Score_adjusted = [S(Q_u, P_u) - S(Q_g, P_u)] / [1 - Sim(P_u, P_g) + ε]
```

其中ε是一个极小值，防止分母为0。

原理解释：如果P_u和P_g极其相似（比如0.9），那么Q_u哪怕只比Q_g的得分高出一点点，也说明它抓住了极难发现的个性化微小特征，因此这个微小的分差会被放大奖励；反之，如果两者差异本来就很大，那么得分门槛就会相应提高。

**方法3：向量空间正交化**

如果我们评估时引入了文本嵌入模型（如DistilBERT等），我们可以在向量空间中直接"剥离"大众特征。

假设P_u的向量为v_u，P_g的向量为v_g。我们可以计算P_u在P_g上的投影，并用P_u减去这个投影，得到一个纯净的、与大众特征完全正交（无关）的用户独有特征向量v_unique：

```
v_unique = v_u - [(v_u · v_g) / (v_g · v_g)] * v_g
```

评估方式：不再让Query和P_u直接计算相似度，而是计算Q_u和Q_g分别与v_unique的相似度。因为v_unique已经剔除了大众重叠部分，Q_g在这上面的得分会断崖式下跌，从而真实反映Q_u的个性化强度。

**方法4：LLM提示词层面的"对比评估"**

既然我们在使用LLM进行打分，比起让LLM单独对(Q,P)组合打分，不如直接在Prompt中引入对比逻辑，让LLM充当一个更苛刻的裁判。

优化前的Prompt："请评估查询Q与画像P的匹配度，打分1-10。"（容易受重叠特征干扰）

优化后的Contrastive Prompt："这里有一个目标用户画像A，和一个大众群体画像B。已知A是B的子集，两者有重叠特征。现在有一个搜索查询Q。请忽略A和B的共同特征，仅针对A独有且区别于B的特征，评估查询Q在多大程度上命中了这些独特特征。请给出1-10的打分，并说明Q具体命中了A的哪些独有特征。"

原理解释：通过自然语言指令（Instruction），强制LLM在注意力机制上忽略公共特征（如商品的基础功能），将权重集中在用户的长尾需求上。

针对我们构建Benchmark Dataset的需求，使用"双重差分法"结合"对比评估Prompt"通常是落地最快且效果最稳健的组合。

我们采用**DiD打分**结合**对比评估**，提供最稳健和可解释的结果。

#### 3.10.3 人类对齐指标

我们计算LLM和人类判断之间的对齐性，使用：
- **Spearman's ρ**：平均分的秩相关
- **Cohen's κ**：偏好选择的一致性
- **Recall@Human**：P(LLM选择个性化 | 人类选择个性化)
- **MAE**：LLM和人类分数之间的平均绝对误差

#### 4.5.6 偏差修正算法

我们形式化双重差分（DiD）评估方法：

**算法：基于DiD的查询评估**

```python
def compute_did_score(personalized_query, public_query, user_profile, public_profile):
    """
    计算查询评估的双重差分（DiD）分数。
    
    参数:
        personalized_query: 生成的个性化搜索查询
        public_query: 基线公开查询
        user_profile: 目标用户的画像/配置文件
        public_profile: 大众群体画像
    
    返回:
        did_score: 量化个性化价值的DiD分数
    """
    # 步骤1：评估所有四个查询-配置文件组合
    s_uu = llm_evaluate(personalized_query, user_profile)
    s_ug = llm_evaluate(personalized_query, public_profile)
    s_gu = llm_evaluate(public_query, user_profile)
    s_gg = llm_evaluate(public_query, public_profile)
    
    # 步骤2：计算偏好
    delta_u = s_uu - s_ug  # 个性化查询的用户偏好
    delta_g = s_gu - s_gg  # 公开查询的用户偏好（基线）
    
    # 步骤3：计算DiD分数
    did_score = delta_u - delta_g
    
    return did_score, {
        's_uu': s_uu, 's_ug': s_ug,
        's_gu': s_gu, 's_gg': s_gg,
        'delta_u': delta_u, 'delta_g': delta_g,
        'did_score': did_score
    }
```

**解释：**
- $\Delta_u$衡量个性化查询比公开查询更偏向用户的程度
- $\Delta_g$衡量任何查询（即使是通用的）在评估为用户画像时的基线优势
- 差值$\Delta_u - \Delta_g$隔离了**增量**个性化价值，完全抵消了基础相似度效应

**理论性质：**

1. **零偏差性质**：当$q_u = q_g$时，$\Delta_u = \Delta_g$，所以$\text{Score}_{\text{DiD}} = 0$
2. **可加分解**：
   $$\text{Score}_{\text{DiD}} = [S(q_u, p_u) - S(q_g, p_u)] - [S(q_u, p_g) - S(q_g, p_g)]$$
   这分离了查询级个性化（$q_u$ vs $q_g$）和配置文件级相似度（$p_u$ vs $p_g$）
3. **单调性**：如果$q_u$比$q_g$捕获更多独特用户维度，$\text{Score}_{\text{DiD}} > 0$
4. **尺度不变性**：该方法对绝对分数差异稳健，专注于相对差异

**实现说明：**

- 我们使用GLM-5作为评估LLM，5分李克特量表（1-5）
- 每个查询-配置文件对独立评估（无对其他查询的引用）
- 基础分数平均过3次独立评估以减少方差
- DiD分数归一化到[0, 1]范围以便跨用户比较

## 5. 讨论

### 5.1 关键发现

我们的实验表明：
1. **接地气画像提高查询质量**：整合技能水平、使用场景和情感配置比仅属性方法产生更真实的查询。
2. **迭代优化是必要的**：单次生成无法捕获细粒度风格特征；多轮优化与特征感知提示是必需的。
3. **16个特征就足够了**：精心策划的风格特征集捕获了必要的语言变化，而没有高维表示的噪声。
4. **靶向噪声注入保持语义**：与随机错误注入不同，难度引导注入在提高真实性的同时保持语义保真度。
5. **LLM评估与人类对齐**：强相关性（ρ=0.82）验证了基于LLM评估的可扩展性。
6. **基础相似度偏差修正揭示真正的个性化价值**：DiD评分将差异化比从1.8×（原始）增加到2.7×，表明传统评估指标系统性低估个性化收益。高画像-大众相似度的用户（>0.9）从偏差修正中受益最大（1.48×放大），因为DiD隔离了埋藏在重叠中的微妙个性化特征。

### 5.2 方法论贡献

我们的工作在查询生成之外做出两个关键的方法论贡献：

**5.2.1 偏差感知评估框架**

我们识别并解决了个性化系统评估中固有的"基础相似度偏差"问题。我们的四种修正方法（DiD、相似度惩罚归一化、向量正交化、对比提示）为不同评估场景提供了互补方法：
- **DiD评分**最适合基准比较，完全消除基础效应
- **向量正交化**无需重新训练即可实现嵌入评估
- **对比提示**在可解释性方面提供实用的部署优势
- **相似度惩罚归一化**为连续评分提供平滑调整

**5.2.2 用户-大众相似度分析**

我们的相关性分析揭示了一个重要洞察：偏差修正的价值随用户-大众画像相似度缩放。对于高重叠的用户（>0.9），传统评估掩盖了高达48%的个性化价值。这表明个性化系统可能在"主流用户"（与大众人口共享许多共同偏好）的评估中被系统性低估。

### 5.3 局限性

1. **领域特异性**：评估限制在艺术与工艺；在其他领域的推广需要验证。
2. **用户冷启动**：流水线需要大量评论历史；新用户可能无法受益。
3. **LLM依赖**：质量受底层LLM能力限制。
4. **计算成本**：12阶段流水线结合迭代优化资源密集。

### 5.4 未来方向

1. **跨领域迁移**：调查跨产品类别的画像迁移。
2. **少样本适应**：为历史有限的用户开发轻量级适应。
3. **实时部署**：针对生产延迟约束优化流水线。
4. **多模态个性化**：扩展到语音查询和视觉搜索。

## 6. 结论

我们介绍了PersonalQuery，这是一个综合性的流水线，用于生成接地气的个性化搜索查询。通过提取细粒度偏好、建模16维语言特征以及采用迭代式优化与特征感知提示，PersonalQuery生成的查询既语义忠实又风格真实。实验结果证明相对于基线方法有显著改进，强人类-LLM对齐验证了我们的评估方法论。拼写难度模型使现实变化注入成为可能，同时保持查询语义。PersonalQuery代表朝着真正个性化搜索体验迈出的重要一步，该体验尊重个人沟通模式，而我们的偏差感知评估框架确保这样的系统可以在研究文献中得到适当的估值。
